Part One · Two Worlds

Why Python feels different.

Before the syntax, the model. Node.js and Python solve the same problems through fundamentally different architectures — and confusing those architectures is what makes Python feel "weird" to a Node developer. Twelve real differences, with the code that shows them. Read this once and the rest of the guide will click.

01 / 12
Runtime Architecture
▲ Node.js

V8 + libuv + a single-threaded event loop. V8 (the same engine as Chrome) executes your JavaScript. libuv (written in C) manages the event loop, thread pool for file I/O, and OS-level async primitives (epoll on Linux, kqueue on macOS, IOCP on Windows).

Your code runs on one main thread. I/O is offloaded to libuv's thread pool or kernel async APIs, and results come back as events on the loop. This is why Node feels fast for I/O — you never block waiting for a socket.

Mental model: a restaurant with one waiter who takes everyone's order, then circles back when each meal is ready.
● Python

CPython is a stack-based bytecode interpreter with no built-in event loop. Your .py file compiles to bytecode (.pyc), and the interpreter executes it instruction by instruction on the main thread.

There's no libuv equivalent baked in. asyncio is a library that adds an event loop on top — you opt into it. Most Python code is synchronous and blocking, and that's the dominant style outside of web servers.

Mental model: a kitchen where you can choose: one cook doing things in order, multiple cooks (threads), or multiple full kitchens (processes).
02 / 12
Default Execution
▲ Node.js

Async is the default. Even reading a tiny file is non-blocking. Synchronous APIs exist (fs.readFileSync) but using them in production is a code smell — they freeze the entire event loop.

// node — non-blocking by default
const data = await fs.readFile("a.txt");
const res  = await fetch("https://api.x");
// these two can interleave with other work
● Python

Sync is the default. open(path).read() blocks the thread until done — and that's fine for most scripts, CLIs, data jobs, and batch work. Async is for when you need concurrency.

# python — sync by default
data = Path("a.txt").read_text()
res  = httpx.get("https://api.x")
# these run one after another, blocking

# to get Node-style concurrency, opt in:
async with httpx.AsyncClient() as c:
    res = await c.get("https://api.x")
03 / 12
Concurrency Models
▲ Node.js

One primary model: event-loop concurrency with async/await. For CPU work, you reach for worker_threads (shared memory possible via SharedArrayBuffer) or cluster (multiple processes). The 95% case is just async functions.

  • 1k concurrent HTTP requests? Trivial, single process.
  • Heavy CPU? Workers or external services.
  • Scale out? cluster or PM2, one process per core.
● Python

Three real models, and you need to know which to use:

  • asyncio — for I/O concurrency in async code (await on many tasks)
  • threading — for blocking I/O in sync code (DB drivers, legacy libs)
  • multiprocessing — for CPU-bound work, true OS-level parallelism

There's also concurrent.futures, a unified API over threads and processes that's often the friendliest entry point.

04 / 12
The GIL
▲ Node.js

No GIL. V8 doesn't share JS objects between worker threads — communication is via message passing (or SharedArrayBuffer for raw bytes). Different model, no equivalent problem.

If you want CPU parallelism, you spawn workers or cluster processes. Done.

● Python

The Global Interpreter Lock: only one thread executes Python bytecode at a time, even on a 16-core CPU. It's a CPython implementation detail dating back to the '90s, kept for C-extension safety and single-threaded speed.

The consequences:

  • 10 threads crunching numbers → slower than 1, due to lock contention
  • 10 threads doing I/O → fine, GIL is released during I/O calls
  • Need real CPU parallelism? Use multiprocessing, drop into C (numpy, polars), or use asyncio for I/O
The future: 3.13 (Oct 2024) introduced an experimental free-threaded build; 3.14 (Oct 2025) made it an officially supported build (PEP 779). It's still opt-in — the GIL is on by default and most of the ecosystem assumes it. Today, plan around it.
05 / 12
Type System
▲ Node.js + TS

TypeScript is a separate compile step. You write .ts, tsc transpiles to .js, types are erased at runtime. Modern runners (tsx, ts-node, bun, deno) hide the step but it's still happening.

Runtime validation needs a separate tool: zod, valibot, io-ts. You write the shape twice unless you derive one from the other.

// shape defined twice
const UserSchema = z.object({
  name: z.string(),
  age:  z.number()
});
type User = z.infer<typeof UserSchema>;
● Python

Types are part of the language (PEP 484, since 2014). No compile step — they're syntax in the .py file itself, ignored at runtime. pyright or mypy checks them statically.

Pydantic uses those same hints for runtime validation. One source of truth, two roles. This is genuinely better than the TS+Zod split.

# shape defined once, used for both
class User(BaseModel):
    name: str
    age:  int

# pyright validates at edit-time,
# pydantic validates at runtime
06 / 12
Memory & GC
▲ Node.js

V8 generational garbage collector — young/old heaps, mark-sweep, mark-compact. Highly tuned for short-lived web workloads. Default heap is ~1.7GB, tune with --max-old-space-size.

You'll rarely think about it until you hit OOM in a long-running process. Profile with Chrome DevTools or --inspect.

● Python

Reference counting + cycle-collecting GC. Most objects die the moment their refcount hits zero — deterministic, predictable, no GC pauses for the common case. A cycle collector runs periodically for circular references.

Two practical implications:

  • Context managers (with) feel natural because cleanup is immediate
  • Memory profile is steadier; fewer GC-pause surprises
07 / 12
Packages & Deps
▲ Node.js

npm/pnpm/yarn + node_modules. Per-project dependency tree on disk. package.json for declared deps, package-lock.json for resolved versions. Tooling is mature; bun and pnpm are fast.

Reality check: a typical Node project pulls 500–2000 transitive dependencies. Supply-chain risk is real (left-pad, event-stream, etc.).

● Python

Virtual environments + pyproject.toml. Each project gets its own isolated Python environment (.venv). Historically fragmented (pip, pipenv, poetry, conda). Today: uv wins — Rust-based, ~10× faster than pip, handles Python versions too.

A typical Python project pulls 20–80 transitive deps. Smaller blast radius. The trade-off: you'll occasionally need a C compiler for native extensions (rare with modern wheels).

The shift to internalize: there is no global install. Always work inside a venv. uv run handles it for you.
08 / 12
Standard Library
▲ Node.js

Lean stdlib, vibrant npm. Built-ins: fs, http, crypto, path, stream, events, util, plus newer node:test and fetch. Anything beyond is npm.

You install packages for date formatting (date-fns), UUIDs (uuid), deep equality (lodash), even is-odd (yes, really).

● Python

"Batteries included" is real. The stdlib is enormous and high quality. You can build full applications with zero external deps.

  • HTTP server + client: http.server, urllib
  • SQLite, JSON, CSV, XML, ZIP, tar — all built-in
  • Dates, UUIDs, hashing, regex, statistics
  • Argument parsing (argparse), logging, threading, multiprocessing
  • Testing (unittest), profiling (cProfile), debugging (pdb)

The first reflex of a Python dev is to check the stdlib. Often it's already there.

09 / 12
Errors & Nullability
▲ Node.js

Two flavors of "missing": null (intentional) and undefined (uninitialized). TS treats them as distinct types. Errors are objects, often subclassed (TypeError, custom classes).

Async errors propagate through promise rejection; unhandled rejections crash the process by default in modern Node.

x?.y?.z          // optional chaining
x ?? "default"    // nullish coalescing
throw new Error("bad");
● Python

One value: None. That's it. Type hints use str | None (or Optional[str]). Exceptions are first-class objects with a rich hierarchy.

Idioms differ: Python prefers EAFP ("easier to ask forgiveness than permission") — try the operation, catch the exception. JS tends toward LBYL ("look before you leap") — check first, then act.

try:
    val = data["key"]
except KeyError:
    val = "default"

# or, the .get() shorthand:
val = data.get("key", "default")
10 / 12
Web Frameworks
▲ Node.js

Express remains the lingua franca (minimal, middleware-based). Fastify for performance and JSON-schema validation. NestJS for opinionated, Angular-style DI. Hono for edge runtimes. Full-stack: Next.js, Remix, SvelteKit.

All work the same way under the hood: middleware chain over the event loop.

● Python

Three real choices, picked by use case:

  • FastAPI — modern, async, type-driven, auto OpenAPI docs. The default pick for new APIs.
  • Django — "batteries included" full-stack: ORM, admin, auth, migrations. Like Rails for Python.
  • Flask — minimal, sync, Express-equivalent. Still everywhere.

FastAPI ↔ Express analogies are surface-level; FastAPI is much closer to tRPC + Zod + Express in spirit.

11 / 12
Production Deploy
▲ Node.js

One node binary, deploy anywhere. Single process is often enough because the event loop saturates I/O easily. Scale via PM2, cluster mode, or just multiple containers behind a load balancer.

Edge-first hosting (Vercel, Cloudflare Workers, Fly) is mature. Cold starts measured in milliseconds.

● Python

You don't run python app.py in production. You run a dedicated server process:

  • uvicorn — ASGI (async) server, used with FastAPI
  • gunicorn — WSGI (sync) process manager, used with Flask/Django
  • Typical: gunicorn -w 4 -k uvicorn.workers.UvicornWorker app:app — 4 worker processes to bypass the GIL

Containers (Docker) are the standard packaging. Cold starts are slower than Node — typically 200ms–1s.

12 / 12
Where Each Shines
▲ Node.js

Real-time I/O, streaming, WebSockets, edge functions. The single-language full-stack story (TS on client and server, shared types) is a genuine advantage.

  • API gateways, BFFs, real-time apps
  • SSR/SSG frameworks (Next, Remix, Astro)
  • Edge compute and serverless functions
  • CLI tools shared with the frontend toolchain
● Python

Data, ML, AI, scientific computing, automation, scripting. The scientific stack (numpy, pandas, polars, scipy, scikit-learn, PyTorch, JAX) has no equivalent anywhere else.

  • ML/AI training and inference (PyTorch, transformers)
  • Data engineering and ETL (Airflow, dbt-core, Dagster)
  • Notebooks and analytics (Jupyter, Polars, DuckDB)
  • DevOps tooling (Ansible, SaltStack), web scraping (Scrapy)
  • "Glue code" between systems — Python's eternal sweet spot
— The takeaway —
Node is "one process, one loop, everything async, npm for everything." Python is "many models to choose from, sync by default, batteries included, pick the right concurrency tool for the job." Once that mental model settles, the rest is syntax — and you already know how to learn syntax.

01

Install uv and make a project // like npm init, but faster

One install gets you Python, virtual envs, and a package manager. Forget the pip/poetry/pyenv discourse — uv replaces all of it.

// terminal
# install (mac/linux)
$ curl -LsSf https://astral.sh/uv/install.sh | sh

$ uv init learn-python && cd learn-python
$ uv run python --version
02

Learn the syntax shifts that bite // indent matters now

The first week you'll write semicolons by accident. That's fine. What actually matters:

  • Indentation is syntax — not style. Four spaces, always.
  • snake_case for variables and functions, PascalCase for classes.
  • None, True, False are capitalized. null doesn't exist.
  • and / or / not instead of && / || / !.
  • No ++. Use i += 1.
// hello.py
# no braces, no semicolons
def greet(name: str) -> str:
    if name and not name.isspace():
        return f"hi, {name}"
    return "hi, stranger"
03

Master the four core types // list, dict, tuple, set

These four cover 95% of data work. list ≈ array, dict ≈ object/Map, tuple is an immutable list, set is what you wish JS had built-in.

// the four horsemen
nums   = [1, 2, 3]              # list (mutable)
user   = {"name": "ada", "age": 30} # dict
point  = (10, 20)               # tuple (immutable)
seen   = {"a", "b", "c"}         # set, O(1) lookup

# access is uniform and lovely
user["name"]        # "ada"
nums.append(4)
"a" in seen        # True
04

Fall in love with comprehensions // .map().filter() but readable

The single most Pythonic thing. Once you can read these, you're halfway to fluent.

// the same operation, three ways
# JS/TS muscle memory
# nums.filter(n => n > 2).map(n => n * 2)

# Python: do it inline
doubled = [n * 2 for n in nums if n > 2]

# dict comprehension
sq = {n: n*n for n in range(5)}

# set comprehension
uniq = {word.lower() for word in text.split()}
05

Ship the week-1 deliverable // prove the basics stick

Write a script that reads a .log file, counts unique IP addresses, and prints the top 5. You'll touch file I/O, dicts, sorting, and string parsing — exactly the foundation everything else builds on.

// the shape of it
from collections import Counter
from pathlib import Path

lines = Path("access.log").read_text().splitlines()
ips   = [line.split()[0] for line in lines]
top   = Counter(ips).most_common(5)

for ip, count in top:
    print(f"{ip:<16} {count}")

01

Type hints + pyright // your TS instincts pay off

Python types look almost identical to TS. The big shift: they aren't enforced at runtime. pyright (from Microsoft, same house as TypeScript) checks them in your editor.

// types are syntax, not runtime
from typing import Optional

def find_user(id: int) -> Optional[str]:
    return users.get(id)   # may be None

# newer syntax (3.10+) reads even better:
def find_user(id: int) -> str | None:
    return users.get(id)
02

Dataclasses & Pydantic // interfaces, but with validation

Two flavors of "shape your data." Dataclasses are stdlib, lightweight — like a TS interface with a constructor. Pydantic adds runtime validation — what zod does, deeply integrated.

// pick the right tool
from dataclasses import dataclass
from pydantic import BaseModel, EmailStr

# lightweight, internal data
@dataclass
class Point:
    x: float
    y: float

# external data (API input, JSON, env)
class User(BaseModel):
    name:  str
    email: EmailStr           # needs pydantic[email]
    age:   int = 0           # default
03

Functions, *args, **kwargs // spread, but better

Python functions have superpowers: keyword arguments, defaults, and variadic collectors. *args collects positional, **kwargs collects keyword.

// the full toolkit
def connect(
    host: str,
    port: int = 5432,           # default
    *,                          # everything after is kw-only
    timeout: float = 10.0,
    **options                   # catch-all kwargs
):
    ...

# callers can be explicit:
connect("db.local", timeout=5, ssl=True)
04

Modules & imports // no node_modules

Each .py file is a module. Folders become packages. The import rules feel weird at first, then click.

  • from math import sqrt — pull one thing.
  • import json — keep the namespace.
  • from .utils import helper — relative import inside a package.
  • Run modules with uv run python -m mypackage, not by file path.

pytest basics // jest, with less ceremony

No describe, no it. Just functions named test_* with plain assert. That's it.

// test_math.py
from mymath import add

def test_add_positives():
    assert add(2, 3) == 5

def test_add_with_zero():
    assert add(0, 7) == 7

01

Context managers // `using`, but everywhere

The with block guarantees cleanup. Files, DB connections, locks, HTTP clients — anything with a lifecycle. You'll use it daily.

// resources, handled
# file closes automatically, even on error
with open("data.json") as f:
    data = json.load(f)

# works with anything that supports it
with httpx.Client() as client:
    r = client.get("https://api.x")
02

Generators & iterators // lazy sequences, native

JS has generators, but in Python they're everywhere — and they're how you process big data without loading it all into memory.

// yield, the magic word
def read_large_file(path):
    with open(path) as f:
        for line in f:        # lazy!
            yield line.strip()

# consumes one line at a time, not the whole file
for line in read_large_file("huge.log"):
    process(line)
03

Decorators // higher-order functions, sugared

That @something above a function? It's just a function that wraps another function. Once you see it, you'll see it everywhere — caching, routing, auth, timing.

// build one, use one
from functools import cache

# stdlib decorator — instant memoization
@cache
def fib(n):
    return n if n < 2 else fib(n-1) + fib(n-2)

# framework decorator — fastapi routing
@app.get("/users/{id}")
def get_user(id: int):
    return db.find(id)
04

Tour the standard library // stop reaching for npm

Python's stdlib is enormous and high-quality. Skim these — you'll use them constantly:

  • pathlib — modern file paths, kills os.path
  • collections — Counter, defaultdict, deque
  • itertools — chain, groupby, pairwise
  • functools — cache, partial, reduce
  • datetime, json, re, subprocess — bread and butter
05

Build a FastAPI service // express, but typed end-to-end

This is where it all comes together. FastAPI uses your type hints to validate input, serialize output, and generate OpenAPI docs — for free.

// main.py — full working API in 15 lines
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class Todo(BaseModel):
    title: str
    done:  bool = False

todos: list[Todo] = []

@app.post("/todos")
def create(todo: Todo) -> Todo:
    todos.append(todo)
    return todo

# setup: uv add "fastapi[standard]"
# run:   uv run fastapi dev main.py
# docs at /docs — auto-generated

01

async / await with httpx // finally familiar territory

This will feel like home. The syntax is identical to TS. The mental model: in Node, async is the default; in Python, you opt in by writing async def and running inside an event loop.

// concurrent fetches
import asyncio, httpx

async def fetch(url: str) -> dict:
    async with httpx.AsyncClient() as c:
        r = await c.get(url)
        return r.json()

async def main():
    # like Promise.all
    results = await asyncio.gather(
        fetch("https://api.x/1"),
        fetch("https://api.x/2"),
    )

asyncio.run(main())
02

The GIL & when concurrency models matter // it's not just async

Python has a Global Interpreter Lock — only one thread runs Python bytecode at a time. Two consequences worth internalizing:

  • I/O-bound work (APIs, DB, files) → use asyncio. The GIL doesn't block I/O.
  • CPU-bound work (number-crunching, image processing) → use multiprocessing, not threads.
  • Or skip the GIL entirely with libraries like numpy and polars that drop into C.
03

Dunder methods // operator overloading, but useful

Methods named __like_this__ hook into Python's data model. They're how custom classes become first-class — printable, comparable, iterable.

// make a class feel native
class Money:
    def __init__(self, amount, currency):
        self.amount = amount
        self.currency = currency

    def __repr__(self):       # print()
        return f"{self.amount} {self.currency}"

    def __add__(self, other):  # the + operator
        return Money(self.amount + other.amount, self.currency)

    def __eq__(self, other):   # the == operator
        return self.amount == other.amount

Part Five · Survival Syntax

Six things with no clean JS equivalent.

These aren't advanced. They're the everyday moves you'll reach for on day one — and the ones a TS reflex won't hand you. Learn these six and you stop writing Python with a JavaScript accent.

Slicing// .slice(), but native & better

Any sequence supports seq[start:stop:step]. Negative indices count from the end. This replaces a dozen array methods.

// slices everywhere
nums = [10, 20, 30, 40, 50]
nums[1:3]     # [20, 30]
nums[-1]      # 50  — last item
nums[:2]      # [10, 20]
nums[::2]     # [10, 30, 50]
text[::-1]    # reverse a string

Unpacking & star// destructuring, leveled up

Tuple unpacking is the most Pythonic assignment. * collects the rest; ** merges dicts; * spreads into calls.

// one assignment, many shapes
a, b = b, a                # swap, no temp
first, *rest = [1, 2, 3, 4] # first=1, rest=[2,3,4]
cfg = {**base, "debug": True} # merge
print(*args)               # spread into a call

enumerate / zip// stop writing range(len(x))

The idiomatic loop helpers. enumerate gives you the index; zip walks two sequences in lockstep. You'll use both constantly.

// loop like a local
# don't: for i in range(len(items))
for i, item in enumerate(items):
    print(i, item)

for name, score in zip(names, scores):
    ...

f-strings// template literals + format specs

Backtick strings, but with a built-in format mini-language. The {x=} form is the fastest debug print you'll ever write.

// formatting, built in
name, pi = "ada", 3.14159
f"hi {name}"       # hi ada
f"{pi:.2f}"        # 3.14
f"{1_000_000:,}"   # 1,000,000
f"{name=}"         # name='ada'  (debug)

if __name__ == "__main__"// wait, where's my entrypoint?

There is no automatic main. Importing a module runs its top level — so real scripts gate their entrypoint behind this guard, or it fires on every import.

// the closest thing to a JS entry file
def main() -> None:
    ...

# runs only when executed directly,
# not when imported by another module
if __name__ == "__main__":
    main()

Ternary & truthiness// ?: reordered, falsy is broader

The ternary reads value-first. Empty collections are falsy — no .length checks. or is like ?? but triggers on any falsy, not just None.

// expressions, not statements
status = "on" if enabled else "off"

# [], {}, "", 0, None  → all falsy
if not items:
    return

name = user_input or "anon"  # note: 0/"" trigger it too

Part Eight · Things That'll Trip You

Nine small gotchas.

— 01 —

Async isn't ambient.

In Node every I/O call is async by default. In Python you opt in. Pick sync or async per project and stick with it.

— 02 —

Types don't run.

Hints are checked by pyright, not at runtime. For runtime validation, that's what pydantic is for.

— 03 —

Mutable defaults bite.

Never write def f(x=[]). The list is shared across calls. Use x=None and initialize inside.

— 04 —

Activate the venv.

No node_modules. Each project gets its own virtual env. uv run handles it for you — use it.

— 05 —

Imports are picky.

Files and folders become modules. The __init__.py question is real. Read up before your first multi-file project.

— 06 —

The GIL exists.

CPU-bound parallelism needs multiprocessing, not threads. I/O-bound work loves asyncio. Know which you have.

— 07 —

is is not ===.

is tests identity (same object), == tests value. Use x is None for null checks — never x == None. For everything else, reach for ==.

— 08 —

No block scope.

Python scoping is function-level. A for/if variable leaks past its block, and closures capture the variable, not its value — the classic loop-in-a-lambda bug.

— 09 —

/ always floats.

5 / 2 is 2.5, not 2 — even for two ints. Want integer division? Use //. This bites every JS dev exactly once.
