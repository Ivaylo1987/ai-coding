1.1

How Claude Decides to Keep Working or Stop

▸
# The loop you must be able to explain:
# 1. Send request to Claude (full conversation history — API is stateless)
# 2. Check stop_reason
#    → "tool_use": execute tools, append results, go to 1
#    → "end_turn": done — present final response
# 3. Tool results go INTO conversation history

async for message in query(prompt="Fix the auth bug",
    options=ClaudeAgentOptions(
        allowed_tools=["Read","Edit","Bash"])):
    if isinstance(message, ResultMessage):
        print(message.result)
3 Anti-Patterns Tested on This Topic ❌ Parsing natural language to detect completion
❌ Using iteration caps as primary stopping mechanism
❌ Checking for text content as completion indicator
Key Detail — Stateless APIThe Messages API is stateless. You must send the full conversation history in every request. Earlier turns don't need to originate from Claude — you can use synthetic assistant messages. The max_tokens parameter is required on every API call — it controls maximum output length, and stop_reason: "max_tokens" is a terminal state alongside "end_turn" and "tool_use".
Model-Driven vs Pre-ConfiguredThe agentic loop is model-driven: Claude reasons about which tool to call next based on context. This is NOT a pre-configured decision tree or hardcoded tool sequence. The model evaluates the current state and determines how to proceed at each step. Don't confuse the loop with a fixed pipeline.
1.2–1.3

How Multi-Agent Systems Communicate and Share Context

▸

The 7 Rules

  • Coordinator manages ALL inter-subagent communication (hub-and-spoke)
  • Subagents have isolated context — NO automatic inheritance from parent
  • Context must be explicitly passed in the subagent prompt
  • Parallel subagents: emit multiple Task calls in ONE coordinator response
  • Coordinator prompts specify goals/criteria, not step-by-step procedures
  • Coordinator's allowedTools must include "Task" to spawn subagents
  • Each subagent is defined via an AgentDefinition — includes description, system prompt, and tool restrictions per subagent type
Exam PatternOutput correct but incomplete? → Check coordinator's task decomposition. "Creative industries" narrowed to only visual arts = decomposition too narrow, not a subagent failure.
1.4–1.5

Guaranteeing Claude Follows Critical Rules Every Time

▸

The Golden Rule

  • Financial operations, identity verification, policy compliance → programmatic enforcement (hooks, prerequisite gates)
  • Prompt instructions have a non-zero failure rate — insufficient for critical business logic
  • PreToolUse: block policy-violating tool calls (refunds > $500)
  • PostToolUse: normalize tool results before model processes them (timestamps, status codes)
This Is Tested RepeatedlyWhenever a question involves financial risk or compliance: if an answer says "enhance system prompt" or "add few-shot examples" — it's a distractor. The answer is programmatic enforcement.

Breaking Down Complex Tasks and Managing Sessions

▸

Two Decomposition Patterns

  • Prompt chaining (fixed sequential) → predictable multi-aspect reviews
  • Dynamic decomposition (adaptive) → open-ended investigation tasks
  • Large code reviews: per-file local passes + cross-file integration pass

Session Management

  • --resume <name> continues named sessions
  • fork_session creates parallel exploration branches
  • Fresh session with injected summary > resuming with stale tool results

2.1

Why Tool Descriptions Are the Most Important Thing You Write

▸

Key Points

  • Descriptions are how the model decides which tool to call — minimal descriptions → wrong tool selection
  • Include: input formats, example queries, edge cases, boundaries vs similar tools
  • When agent misroutes: expand descriptions first (lowest effort, highest leverage)
  • Rename tools to eliminate functional overlap (e.g., analyze_content → extract_web_results with a web-specific description)
  • Split generic tools into purpose-specific ones with defined I/O contracts (e.g., split analyze_document into extract_data_points, summarize_content, and verify_claim_against_source)
  • Review system prompts for keyword-sensitive instructions that might create unintended tool associations — a keyword in the system prompt can override well-written tool descriptions
2.2

2.3–2.5

Which Tools to Give Each Agent and How to Control Selection

▸

tool_choice Options (memorize these)

  • "auto" — model may return text OR call a tool
  • "any" — MUST call a tool, can choose which (guarantees structured output)
  • {"type":"tool","name":"..."} — forced specific tool call

Built-in Tool Selection (memorize this)

  • Grep = search file contents · Glob = find files by name/path
  • Read = load file · Write = create/overwrite · Edit = targeted unique-text match
  • Edit fails on non-unique match? → Read + Write as fallback
  • Exploration pattern: Grep → Read → DON'T read all files upfront

MCP Configuration

  • Project: .mcp.json (shared, VCS) with ${ENV_VAR} expansion
  • User: ~/.claude.json (personal/experimental)
  • MCP resources expose content catalogs (issue summaries, DB schemas, documentation hierarchies) to give agents visibility into available data without requiring exploratory tool calls
  • Prefer existing community MCP servers for standard integrations (e.g., Jira) — reserve custom servers for team-specific workflows

3.4–3.6

When to Plan vs Execute, and Running Claude in CI Pipelines

▸

Plan Mode vs Direct Execution

  • Plan mode: multi-file, architectural decisions, multiple approaches, unfamiliar codebases
  • Direct execution: single-file fix, one validation check, well-understood scope
  • If task description mentions restructuring/migration → start in plan mode (complexity is stated)
  • Use the Explore subagent for verbose discovery phases — isolates exploration output from main context, preventing context window exhaustion during multi-phase tasks
# CI/CD — the pattern tested directly
claude -p "Analyze this PR" --output-format json --json-schema schema.json
# -p = non-interactive (prevents hang)
# Session isolation: don't review code in the same session that generated it

CI/CD Key Details

  • Session context isolation: an independent review instance catches more issues than the same session that generated the code
  • Include prior review findings in context when re-running reviews after new commits — instruct Claude to report only new or still-unaddressed issues to avoid duplicate comments
  • Provide existing test files in context so test generation avoids suggesting duplicate scenarios already covered
  • Document testing standards and available fixtures in CLAUDE.md for CI-invoked Claude Code

Iterative Refinement Patterns

  • 2–3 concrete I/O examples when prose is ambiguous
  • Test-driven: write tests first → share failures → iterate
  • Interview pattern: Claude asks questions before implementing
  • Interacting issues → single message · Independent → sequential

4.1–4.2

Writing Prompts That Reduce False Positives

▸

Key Principles

  • "Flag when claimed behavior contradicts code" >> "check comments are accurate"
  • "Be conservative" does NOT improve precision — use specific categorical criteria instead
  • Define explicit severity criteria with concrete code examples for each severity level to achieve consistent classification across reviewers
  • Temporarily disable high false-positive categories to restore developer trust while you improve prompts for those categories — high FP in one area undermines confidence across all categories
  • Few-shot: 2–4 targeted examples for ambiguous scenarios showing reasoning
  • Few-shot enables generalization to novel patterns (not just matching examples)
  • Few-shot is the most effective technique for reducing hallucination in extraction tasks (e.g., handling informal measurements, varied document structures)
4.3–4.4

Getting Reliable Structured Data from Claude

▸

The situation: You need Claude to extract data from invoices, contracts, or documents and return it as clean JSON that downstream systems can process — not freeform text. The exam tests whether you know how to guarantee the output format, handle missing data gracefully, and fix extraction errors.

Forcing Structured Output (Scenario 6: Data Extraction)

  • Use tool_use with a JSON schema to force Claude to return data in an exact structure. This eliminates JSON syntax errors (missing brackets, wrong types) — Claude must match your schema.
  • But schema compliance does NOT prevent semantic errors — Claude might put the vendor name in the wrong field, or line item totals that don't add up. You still need validation.
  • Use Pydantic (Python) or JSON schema validation after extraction to catch these semantic errors that tool_use alone can't prevent.

Designing Schemas That Prevent Fabrication

  • Make fields nullable when the source document might not contain that data. If you mark a field as required and it's not in the document, Claude will fabricate a value to satisfy the schema.
  • Add an "unclear" option to enum fields for genuinely ambiguous cases — gives Claude an honest escape instead of forcing a wrong choice.
  • Use "other" + a detail string field for extensible categories — when the real value doesn't fit your predefined options.
  • Include format normalization rules in your prompt alongside the schema (e.g., "all dates must be ISO 8601", "currency as numbers without symbols") — source documents use inconsistent formats and the schema alone won't normalize them.

When Extraction Fails: Retry vs Give Up

  • Retry works for format/structural errors: send the document again with the failed extraction AND the specific validation error. Claude can self-correct when it knows what went wrong.
  • Retry is useless when the information simply isn't in the source document — no amount of re-reading will create data that doesn't exist.
  • For ongoing quality: add a detected_pattern field to each finding so you can analyze which patterns trigger false positives when reviewers dismiss results.
  • For self-checking: have Claude extract "calculated_total" alongside "stated_total" to flag discrepancies. Add "conflict_detected": true when sources contradict.
4.5–4.6

Batch Processing for Cost Savings and Independent Review for Quality

▸

Two separate concepts tested together: (1) The Message Batches API lets you process hundreds of documents at 50% off — but only when you can wait for results. (2) When Claude reviews its own code, it has blind spots — an independent review instance catches more bugs. The exam tests whether you know when each applies.

Batch Processing — When and Where to Use It (Scenario 5: CI/CD, Scenario 6: Extraction)

  • The Message Batches API gives 50% cost savings but takes up to 24 hours with no latency guarantee.
  • Use batch for: overnight technical debt reports, weekly security audits, nightly test generation — anything where the results can wait until tomorrow.
  • Never use batch for: blocking pre-merge checks, real-time user-facing operations — developers can't wait 24 hours to merge a PR.
  • Batch does NOT support multi-turn tool calling within a single request — each batch item is a standalone one-shot call.
  • Before batching 1,000 docs: refine your prompt on a sample set of 20–30 first. This maximizes first-pass success and avoids expensive resubmission cycles.
  • Track failures using custom_id on each request. When some fail, resubmit only those with modifications (e.g., chunk oversized documents that exceeded context limits).

Independent Review — Why Claude Shouldn't Grade Its Own Work (Scenario 5: CI/CD)

  • When Claude generates code and then reviews it in the same session, it retains its reasoning context. It's less likely to question decisions it just made — like a student grading their own test.
  • A separate Claude instance without the generation context catches subtle issues that self-review misses. This is tested directly in the exam.
  • For large PRs (10+ files): don't review everything in one pass. Split into per-file local analysis (consistent depth on each file) plus a cross-file integration pass (catches data flow issues between files). Single-pass reviews produce contradictory feedback — flagging a pattern as bad in one file while approving it in another.
Exam Trap — "Switch Both to Batch"A manager proposes batch for both pre-merge checks AND overnight reports to save 50%. The correct answer is always: batch for overnight only, keep real-time for blocking checks. The savings don't justify breaking developer workflow.

5.1

Keeping Critical Information Alive in Long Conversations

▸

3 Problems + Solutions

  • Summarization loss: numbers/dates condensed → Extract "case facts" block outside summarized history
  • Lost in the middle: middle sections missed → Place key findings at beginning + section headers
  • Token bloat: 40+ fields per tool result → Trim to relevant fields before accumulation
  • API is stateless: pass complete conversation history in every subsequent request to maintain coherence
  • Modify upstream agents to return structured data (key facts, citations, relevance scores) instead of verbose content when downstream agents have limited context budgets
5.2

When to Hand Off to a Human (and When Not To)

▸

Escalate Immediately When:

  • Customer explicitly requests a human (don't investigate first)
  • Policy gaps or exceptions
  • Multiple customer matches → ask for identifiers, don't use heuristics
  • Inability to make meaningful progress after attempting resolution
The Frustration PatternWhen a customer is frustrated but the issue is within the agent's capability: acknowledge the frustration while offering resolution. Only escalate if the customer reiterates their preference for a human. Don't escalate purely based on negative sentiment — frustration ≠ complexity.
Unreliable Proxies (the exam tests these) ❌ Sentiment analysis (frustration ≠ complexity)
❌ Self-reported confidence (model is confident on hard cases too)
✅ Explicit criteria + few-shot examples in system prompt
5.3–5.6

Handling Failures, Exploring Codebases, and Tracking Sources

▸

Error Propagation

  • Structured context: failure type, attempted query, partial results, alternatives
  • Subagents: local recovery for transient errors, propagate only unresolvable
  • Synthesis output should include coverage gap annotations — indicate which findings are well-supported vs which topic areas have gaps due to unavailable sources
  • ❌ Generic "search unavailable" · ❌ Empty results as success · ❌ Kill entire workflow on one failure

Codebase Exploration

  • Context degrades in extended sessions — models start giving inconsistent answers and referencing "typical patterns" rather than specific findings
  • Scratchpad files persist findings across context boundaries
  • Subagents isolate verbose exploration · /compact when context fills
  • Crash recovery: each agent exports structured state to a known location (manifests), and the coordinator loads these on resume and injects them into agent prompts

Information Provenance

  • Require subagents to output structured claim-source mappings (source URLs, document names, relevant excerpts) that downstream agents preserve through synthesis
  • Conflicting stats → annotate with both values and source attribution, don't arbitrarily select one
  • Require publication/collection dates in structured outputs to prevent temporal differences from being misinterpreted as contradictions
  • Render different content types appropriately — financial data as tables, news as prose, technical findings as structured lists

Human Review & Confidence Calibration

  • Aggregate accuracy (e.g., 97%) may mask poor performance on specific document types or fields — validate accuracy by segment before reducing human review
  • Use stratified random sampling of high-confidence extractions for ongoing error rate measurement and novel pattern detection
  • Have models output field-level confidence scores, then calibrate review thresholds using labeled validation sets
  • Route extractions with low confidence or ambiguous/contradictory sources to human review, prioritizing limited reviewer capacity

  3. Add 3 few-shot examples showing extraction from varied formats: narrative paragraph, structured table, and a document with inline footnotes. Measure consistency improvement.
  4. Design a batch strategy: simulate 100 documents via the Message Batches API concept. Track which fail by custom_id. Calculate: if batch takes up to 24 hrs and SLA is 30 hrs, what's the latest you can submit?
  5. Implement confidence routing: have the model output field-level confidence. Route low-confidence to human review. Check: does the 97% overall accuracy hold across all document types, or does one type drag it down?

Project 4: Multi-Agent Research Pipeline

Domains: D1, D2, D5 · Maps to: Scenario 3

  2. Implement parallel execution: coordinator emits 2 Task calls in one response. Measure latency vs sequential. Log message types: AssistantMessage → TaskProgress → TaskNotification.
  3. Design structured output: each finding = {claim, evidence_excerpt, source_url, source_name, publication_date, confidence}. Verify the synthesis agent preserves all source attributions when combining.
  4. Simulate a timeout: make one subagent return structured error (failure_type: "timeout", attempted_query: "...", partial_results: [...], alternatives: [...]). Verify coordinator proceeds with partial results and annotates gaps in the final report.
  5. Test conflicting sources: two credible sources report different statistics. Verify synthesis preserves both values with full attribution and distinguishes "well-established" from "contested" findings.

The 7 Anti-Patterns — Know What NOT To Do

Community exam-takers report that many wrong answers are architectural anti-patterns engineers commonly reach for before understanding production implications. Memorize all 7.

Anti-Pattern #1: Prompt-based enforcement for critical business logic Wrong: "Enhance system prompt to require identity verification before refunds."
Right: Programmatic prerequisite gate that blocks process_refund until get_customer returns verified ID.
Why: Prompts have a non-zero failure rate. Financial operations need deterministic guarantees.
Anti-Pattern #2: Confidence-based escalation Wrong: "Have the agent self-report confidence 1-10, auto-escalate below threshold."
Right: Explicit escalation criteria with few-shot examples in system prompt.
Why: LLM self-reported confidence is poorly calibrated — the agent is incorrectly confident on hard cases.
Anti-Pattern #3: Generic error responses Wrong: Returning "Operation failed" or empty results as success.
Right: Structured error with errorCategory, isRetryable, description, and partial results.
Why: Generic errors hide the context needed for intelligent recovery by the coordinator.
Anti-Pattern #4: Bigger model for attention problems Wrong: "Switch to higher-tier model with larger context window for 14-file review."
Right: Split into per-file local passes + cross-file integration pass.
Why: Larger context windows don't solve attention quality issues. Focused passes do.
Anti-Pattern #5: Batch API for blocking workflows Wrong: "Switch pre-merge checks to Message Batches API for 50% savings."
Right: Batch for overnight/weekly only. Keep real-time for blocking pre-merge checks.
Why: Batch has up to 24hr processing with no latency SLA. Devs can't wait to merge.
Anti-Pattern #6: Self-review in the same session Wrong: "Have the same Claude instance review its own generated code."
Right: Independent review instance without prior reasoning context.
Why: Model retains reasoning context from generation, less likely to question own decisions.
Anti-Pattern #7: Over-provisioning agent tools Wrong: "Give synthesis agent access to all web search tools for self-service verification."
Right: Scoped verify_fact tool for simple lookups; complex verification through coordinator.
Why: 18 tools degrades selection vs 4-5 focused tools. Agents misuse tools outside their specialization.
